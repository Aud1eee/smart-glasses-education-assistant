## 1. 实验目的

本实验的目标不是证明系统能够直接获得学生的“真实专注真值”，而是验证当前 **Learning State Guardian** 原型在已知模拟学习场景下，是否能够：

1. 稳定输出符合预期的学习状态标签；
2. 在完整 demo 流程中保持一致的状态统计结果；
3. 为后续接入 Rokid 眼镜真实输入提供一个可重复、可对照的算法验证基线。

因此，本实验属于**原型一致性验证**与**演示级算法行为验证**，而不是大规模真实用户实验。

## 2. 实验对象

本次验证对象为项目中的学习状态感知后端模块，主要包括：

- 姿态相关状态估计：
  - `Behavioral_Alignment`
  - `Cognitive_Load`
  - `Fatigue_Risk`
  - `Confidence_Level`
- 自适应调节引擎：
  - `Focus settling`
  - `Stable focus`
  - `Load rising`
  - `High load`
  - `Regulate now`
  - `Fatigue risk`
- 难点事件标记与 demo 流水线。

## 4. 实验设计

### 4.1 场景级验证

为了验证算法在不同学习状态下的行为是否合理，系统构造了 4 类确定性模拟场景：

| 场景 | 含义 | 预期输出特征 |
| --- | --- | --- |
| `stable` | 学习状态稳定 | 负荷低、专注代理分高，最终状态趋向 `Stable focus` |
| `rising` | 认知负荷逐步上升 | 负荷进入中高区间，状态应出现 `Load rising` 或进一步调节信号 |
| `overload` | 长时间高负荷 | 负荷高、专注代理分下降，状态进入 `High load / Regulate now / Fatigue risk` |
| `recovery` | 高负荷后恢复 | 负荷下降、专注代理分回升，最终重新回到稳定学习状态 |

需要说明的是，系统当前存在一个保守的过渡态 `Focus settling`。  
因此在 `stable` 和 `recovery` 场景中，只要尾段主导状态为 `Focus settling` 或 `Stable focus`，都视为合理结果。这是因为调节引擎有意保留了一个“重新稳定”的缓冲区，而不是在状态刚恢复时立刻切换为完全稳定。

### 4.2 流水线级验证

除单场景验证外，还对完整 demo 流水线进行了端到端检查。该流程会自动生成：

- [demo_study_report.csv](</C:/Users/11721/Desktop/focus_project_windows/data/demo_study_report.csv>)
- [demo_difficulty_events.csv](</C:/Users/11721/Desktop/focus_project_windows/data/demo_difficulty_events.csv>)

并对以下指标进行检查：

- 样本数是否足够形成完整时间轴；
- 平均行为对齐度是否处于合理范围；
- 平均认知负荷是否呈现“稳定 + 负荷上升 + 恢复”的混合特征；
- 平均疲劳风险是否存在但不过度主导；
- 高负荷占比、漂移风险占比、低置信度占比是否合理；
- 是否成功产生至少一个难点事件。

## 5. 评价指标

本次实验主要使用以下指标：

- `avg_alignment`：平均行为对齐度
- `avg_focus`：平均专注代理分
- `avg_load`：平均认知负荷
- `avg_fatigue`：平均疲劳风险
- `drift_ratio`：漂移风险占比
- `high_load_ratio`：高负荷占比
- `low_conf_ratio`：低置信度占比
- `difficulty_event_count`：难点事件数量

这些指标并不直接代表学生真实的心理状态，而是用于验证当前算法框架在模拟学习过程中的**内部一致性**与**行为合理性**。

## 6. 实验结果

### 6.1 场景级结果

当前验证结果显示，4 类确定性场景全部通过：

| 场景 | 尾段主导状态 | 最终状态 | 平均负荷 | 平均专注代理分 | 结果 |
| --- | --- | --- | ---: | ---: | --- |
| Stable focus | Focus settling | Stable focus | 1.2 | 98.8 | PASS |
| Rising cognitive load | Fatigue risk | Fatigue risk | 55.2 | 49.3 | PASS |
| High-load regulation | Fatigue risk | Fatigue risk | 85.7 | 23.4 | PASS |
| Recovery state | Focus settling | Stable focus | 5.0 | 95.1 | PASS |

从结果可以看出：

- `stable` 与 `recovery` 场景的平均负荷均较低，平均专注代理分较高，最终都回到 `Stable focus`；
- `rising` 场景的平均负荷为 `55.2`，说明系统能够识别中高水平的调节压力；
- `overload` 场景的平均负荷为 `85.7`，平均专注代理分仅为 `23.4`，说明系统在高压力输入下能明显区分异常学习状态。

### 6.2 端到端 demo 流水线结果

完整 demo 流水线检查结果为 `8/8 passed`，总体状态为 `PASS`。

当前主要统计结果如下：

- 样本数：`349`
- 平均行为对齐度：`63.9`
- 平均专注代理分：`61.7`
- 平均认知负荷：`42.6`
- 平均疲劳风险：`30.6`
- 漂移风险占比：`57.0%`
- 高负荷占比：`36.7%`
- 低置信度占比：`0.0%`
- 难点事件数量：`1`

这说明当前 demo 流程既包含稳定学习段，也包含明显的负荷升高段与恢复段，因此能够较完整地展示系统主线逻辑：

**状态感知 -> 实时调节 -> 难点标记 -> 课后复盘**

## 7. 实验结论

基于当前验证结果，可以得到以下结论：

1. 当前学习状态感知原型在 4 类标准模拟场景下均能输出符合预期的状态变化；
2. 完整 demo 流水线在样本数、负荷分布、疲劳风险、事件数量等方面均通过检查；
3. 因此，该原型在**演示级验证**与**算法一致性验证**层面具有较好的稳定性；
4. 这为后续将模拟输入替换为 Rokid 眼镜真实姿态/视觉输入提供了可靠的实验基线。

## 8. 局限性说明

尽管当前验证结果通过，但仍需明确以下局限：

1. 当前实验主要基于确定性模拟数据，而不是真实课堂中的大规模用户数据；
2. 当前状态判断仍以姿态相关特征为主，尚未形成真正的多模态学习状态估计系统；
3. 实验结果反映的是**模型行为一致性**，而不是对真实学习心理状态的绝对测量；
4. 在当前 Windows runtime bridge 下，`matplotlib` 存在二进制兼容问题，因此本次验证允许“统计结果有效但热力图重导出跳过”这一情况存在。

## 9. 后续实验建议

后续若继续完善毕业设计，建议增加以下验证层次：

1. **Rokid 真实输入验证**
   使用实际设备姿态流替换当前模拟器输入；
2. **人工标注对照**
   对学习视频片段进行“稳定学习 / 负荷上升 / 疲劳 / 恢复”等人工标注；
3. **任务表现对照**
   将状态结果与小测成绩、复盘命中率、任务完成情况进行关联；
4. **多模态扩展验证**
   后续接入眼部或面部特征后，再验证行为对齐、认知负荷和疲劳风险三类信号的联合表现。
